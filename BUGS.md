## BUG-001 ŌĆö Exercise name shows as "Exercise" after restoring from background

**Severity:** P0 Critical

**Description:**
When the user starts a workout (from a routine or manually with added exercises), backgrounds the app (or device kills the process), then returns, all exercise cards show the placeholder text "Exercise" instead of the actual exercise name (e.g., "Bench Press"). The exercise name cannot be recovered from local state.

**Root cause:**
`WorkoutExercise` has `@JsonKey(includeToJson: false)` on the `exercise` field:

- `lib/features/workouts/models/workout_exercise.dart` line 18
- `lib/features/workouts/models/workout_exercise.g.dart` lines 22-30 (generated toJson omits the `exercise` key)

`ActiveWorkoutState` is serialized to JSON and persisted to Hive in `WorkoutLocalStorage`. On restore, `WorkoutExercise.fromJson` reads back `exercise: null` because the key was never written. The UI at `lib/features/workouts/ui/active_workout_screen.dart` line 575 falls back to `'Exercise'`:

```dart
exercise?.name ?? 'Exercise'
```

**Steps to reproduce:**
1. Start a workout from any routine (or add an exercise manually).
2. Background the app for long enough for the process to be killed, OR restart the app without finishing the workout.
3. Observe exercise card headers show "Exercise" instead of the exercise name.

**Suggested fix:**
Change `@JsonKey(includeToJson: false)` to `@JsonKey(includeToJson: true)` on `WorkoutExercise.exercise`, then re-run `make gen`. The exercise data is required for correct display after restoration. The `exercise` field is already read correctly by the generated `fromJson` (line 17-19 of workout_exercise.g.dart).

## BUG-002 ŌĆö Form tips display literal `\n` instead of line breaks for default exercises

**Severity:** P1 High

**Description:**
On the exercise detail screen (and the active workout exercise info sheet), form tips for all seeded default exercises render as a single block of text with literal `\n` characters ŌĆö e.g., `"Plant feet flat on the floor...\nLower the bar..."` ŌĆö instead of as separate bulleted tips.

**Root cause:**
The SQL migration `supabase/migrations/00010_seed_exercise_descriptions.sql` stores the separator as the two-character sequence `\n` inside standard SQL single-quoted strings. In PostgreSQL with `standard_conforming_strings = on` (the default in modern PostgreSQL), `'\n'` inside single quotes is **not** interpreted as a newline (U+000A) ŌĆö it is stored literally as a backslash followed by the letter `n`.

The Flutter widget `ExerciseFormTipsSection` in `lib/shared/widgets/exercise_info_sections.dart` line 58 splits on `'\n'` (actual newline U+000A, `String.split('\n')`). Since the stored data contains `\n` (two chars), the split finds no delimiters and the entire string renders as one tip with visible backslash-n characters.

The existing widget test at `test/widget/shared/widgets/exercise_info_sections_test.dart` line 111-124 passes Dart string literals with `\n` (actual newlines), so it does **not** reproduce the database content mismatch.

**Steps to reproduce:**
1. Open any default exercise detail screen (e.g., "Barbell Bench Press").
2. Observe form tips show as a single block: `"Plant feet flat...\nLower the bar...\nPress up..."`.

**Suggested fix (two-part):**
- **Option A (preferred):** Change the SQL migration to use PostgreSQL escape strings. Replace each `form_tips = '...'` with `form_tips = E'...'`. The `E''` prefix enables escape processing so `\n` becomes a real newline. Example:
  ```sql
  form_tips = E'Plant feet flat on the floor...\nLower the bar...'
  ```
- **Option B:** Change the Flutter widget to additionally split on the literal two-character sequence `\n`. Update `ExerciseFormTipsSection.split('\n')` to `split(RegExp(r'\n|\\n'))`.

## BUG-003 ŌĆö Starting a routine with exercises fails silently when exercise resolution failed

**Severity:** P1 High

**Description:**
If `RoutineRepository._resolveExercises` fails to match any exercise IDs (e.g., network error, stale cache, or the exercises table fetch silently returns empty), every `RoutineExercise.exercise` field remains `null`. The `startRoutineWorkout` action in `lib/features/routines/ui/start_routine_action.dart` lines 43-44 filters out all exercises with `null` exercise references:

```dart
final exercises = routine.exercises
    .where((re) => re.exercise != null && re.exercise!.deletedAt == null)
```

If all exercises are filtered out, the guard at line 60 (`if (exercises.isEmpty) return;`) silently returns ŌĆö the workout never starts and no error is shown to the user.

**Root cause:**
The filter is correct as a safety guard, but there is no user feedback when all exercises are filtered out. Additionally, the `_resolveExercises` method in `lib/features/routines/data/routine_repository.dart` line 114 returns the original unresolved list without errors if `exerciseMap.isEmpty` (early return on line 114), meaning partial or full resolution failures are silent.

**Steps to reproduce:**
1. Create a routine with exercises.
2. Simulate a condition where exercises fail to resolve (e.g., temporary network failure between fetching the routine list and starting the workout).
3. Tap "Start" on a routine ŌĆö nothing happens, no error message.

**Suggested fix:**
Add a user-facing error snackbar in `start_routine_action.dart` when `exercises.isEmpty`:
```dart
if (exercises.isEmpty) {
  ScaffoldMessenger.of(context).showSnackBar(
    const SnackBar(content: Text('Could not load exercises. Please try again.')),
  );
  return;
}
```

## BUG-004 ŌĆö Routine start uses weight=0 for first-time exercises instead of smart defaults

**Severity:** P2 Medium

**Description:**
When starting a workout from a routine and there is no previous session data for an exercise (first time ever logging it), all sets are pre-filled with `weight: 0`. The manual "Add Exercise" path uses `defaultSetValues(equipmentType, weightUnit)` to provide sensible starting weights (e.g., 20 kg for barbell, 10 kg for dumbbell). The routine start path skips this entirely.

**Root cause:**
In `lib/features/workouts/providers/notifiers/active_workout_notifier.dart`, the `startFromRoutine` method at lines 106-125 uses:
```dart
weight: prev?.weight ?? 0,
```
There is no fallback to `defaultSetValues` when `prev` is null. The `startWorkout` ŌåÆ manual `addSet` path in `active_workout_screen.dart` lines 730-737 does use equipment-type defaults, creating an inconsistent user experience.

**Steps to reproduce:**
1. Create a new exercise (never logged before).
2. Add it to a routine.
3. Start a workout from that routine.
4. All sets show weight `0` instead of the equipment-type default (e.g., `20 kg` for a barbell exercise).

**Suggested fix:**
In `startFromRoutine`, after `final sets = List.generate(...)`, use `defaultSetValues` as the final fallback:
```dart
weight: prev?.weight ?? defaultSetValues(re.exercise.equipmentType, weightUnit).weight,
reps: re.targetReps ?? prev?.reps ?? defaultSetValues(re.exercise.equipmentType, weightUnit).reps,
```
This requires the `weightUnit` to be available in the notifier (e.g., injected via the profile provider).

## BUG-005 ŌĆö `_resolveExercises` returns silently when exerciseMap is empty

**Severity:** P2 Medium

**Description:**
In `RoutineRepository._resolveExercises` at line 114:
```dart
if (exerciseMap.isEmpty) return routines;
```
This early-return path returns routines without resolving `exercise` references ŌĆö with all `RoutineExercise.exercise` fields still `null`. This happens when the exercises batch-fetch returns no rows (e.g., all exercises were soft-deleted, network error, or ID mismatch). Callers receive routines that appear valid but have no embedded exercise data. This silently feeds into Bug-003.

**Root cause:**
`lib/features/routines/data/routine_repository.dart` line 114. The condition checks `exerciseMap.isEmpty` to skip an unnecessary loop, but it silently swallows a potentially broken state.

**Note:** This is only a problem if the exercise fetch itself fails or returns empty unexpectedly. The intent was a cheap optimization (skip the loop if no exercises exist), but the early return should only apply when `ids.isEmpty` (no exercises in any routine), not when the fetch returned empty.

**Suggested fix:**
Remove the early return at line 114 and let the `map` proceed ŌĆö it handles empty maps gracefully (all lookups return null, exercises stay null, BUG-003 mitigation catches it). Or, separate the empty-IDs guard from the empty-fetch result:
```dart
// Already guarded in _fetchExerciseMap: if (ids.isEmpty) return {};
// Remove: if (exerciseMap.isEmpty) return routines;
```

## BUG-006 ŌĆö Active workout screen "Add Set" button: warmup-to-working fallback has unreachable code path

**Severity:** P3 Low

**Description:**
In `lib/features/workouts/ui/active_workout_screen.dart` lines 710-717, the code inside the `else if (prevSet.setType != SetType.warmup)` block's `else` branch attempts to re-check `lastSetForNewRow != null` ŌĆö but `lastSetForNewRow` was already checked and found `null` to reach this branch (because it falls into `else if` after `if (lastSetForNewRow != null)` at line 702). This inner check at line 713 is always false and dead code.

**Root cause:**
Lines 710-727 of `active_workout_screen.dart`:
```dart
} else if (activeExercise.sets.isNotEmpty) {
  final prevSet = activeExercise.sets.last;
  if (prevSet.setType != SetType.warmup) {
    defaultWeight = prevSet.weight;
    defaultReps = prevSet.reps;
  } else if (lastSetForNewRow != null) {  // ŌåÉ always false here
    defaultWeight = lastSetForNewRow.weight;
    ...
  } else {
    // equipment defaults
  }
}
```
When execution reaches `else if (prevSet.setType != SetType.warmup)`, we are already past the `if (lastSetForNewRow != null)` guard at line 702. So `lastSetForNewRow` is guaranteed to be null. The `else if (lastSetForNewRow != null)` branch is dead code and the comment "Warmup -> working: prefer previous session data" is misleading. Equipment defaults are always used for warmup-to-working transitions.

**Steps to reproduce:**
1. Have no previous workout data for an exercise.
2. Add a warmup set and complete it.
3. Tap "Add Set" ŌĆö the next set will use equipment defaults (correct behavior) but via the dead-code path.

**Suggested fix:**
Remove the dead `else if (lastSetForNewRow != null)` branch and its body. The remaining `else { equipment defaults }` is the correct fallback. This is a logic clarity issue, not a user-visible defect.
